# 阶段10A：认证鲁棒性测试与异常流量模拟

阶段10A已经补齐开发指南第21、22和26.6节要求的软件能力。系统在封闭局域网中使用独立测试端完成Message冲突副本、延迟重复报文和高频无效流量模拟；发送侧故障注入用于丢包、逻辑断链和固定延迟。控制协议、节点协议和UDP认证报文序列化保持独立，测试端只接收公开上下文，不接收Sender种子、未披露密钥或完整原始载荷。

## 已实现功能

- `AttackControl`提供独立TCP控制消息：公开上下文、`std::variant`模式专用计划、计划确认、Ready、统一启动、停止、紧急停止和模式专用执行状态。公开上下文包含密码算法、认证模式、报文数量、每间隔发包数、间隔毫秒数、披露延迟和改进模式分组参数，但不包含任何私有密钥或业务载荷。
- 计划参数在独立测试端配置并返回管理端；管理端只校验当前轮次、目标Sender和chain，不替测试端生成Message变化内容。
- 测试端GUI支持报文编号列表/范围、Message字节位置、XOR掩码、重复次数、延迟、重复间隔、PPS、持续时间和报文大小，并实时标红非法输入。超过改进TESLA检测阈值时必须明确确认。
- 测试端只捕获目标Sender实际IP、目标chain且能够按公开上下文完整解码的TESLA UDP报文。Message冲突副本只改变32B Message中的指定字节；延迟重复报文保留完整原始UDP负载。
- 高频无效流量只能发送到程序内部固定TESLA组播地址和端口，用户不能输入任意目标。
- 非高频无效流量计划由管理端向持有目标Sender上下文的全部Receiver安装临时来源映射；所有Receiver确认后才向测试端返回计划接受。映射绑定round、Sender、chain、正常源IP、测试端IP和过期时间。
- Manager使用一个未来时间戳启动测试端和全部认证节点。若节点启动下发失败，测试端会收到回滚停止；测试端或Receiver断线时计划取消或紧急停止并清理映射。
- Receiver维护有界候选版本和去重信息，区分正常来源、测试端来源和未知来源；停止轮次时自动删除临时来源映射。
- 发送侧故障注入支持按随机种子复现的丢包、逻辑断链和固定延迟。丢包和逻辑断链始终保护标签承载的组末报文；原生/改进对比可显式使用相同的标签保护分组。
- PC节点和无人机监视端可导出逐包观测、结构化失败和1000ms无效流量汇总CSV/JSON；无人机监视端可导出验证耗时、硬件计数状态、估算能耗及通信开销原始记录；测试端可导出轮次摘要和本地逐次执行记录。
- 节点逐轮归档由公共配置和Sender/Receiver角色专用`std::variant`详情组成，不使用包含所有无关字段的平铺领域结构；CSV只在文件格式层按第29.5节展开为固定列。
- CMake自动写入构建所对应的Git提交；工作树不干净时追加`-dirty`。文本和文件载荷统一记录真实SHA-256，64位标识、时间戳和计数在JSON中使用字符串保存。

## 逐轮原始数据

PC节点和无人机监视端的认证监视页面提供“导出逐轮CSV”和“导出逐轮JSON”。每条归档包含`experimentId`、`runId`、时间戳、Git提交、节点和Sender身份、chain、认证模式、密码算法、载荷SHA-256、调度参数、故障配置、收发/认证/失败/缺失数量、回退组数量、验证耗时、认证字节、估算能耗、文件大小与恢复Hash、轮次状态以及无效样本原因。

Sender在接受载荷时计算SHA-256；Receiver只在实际恢复出文本或文件后计算，未完成恢复的Receiver记录允许留空。发送侧故障细节不会下发给Receiver，因此Receiver导出使用`NOT_DISTRIBUTED`，不猜测成`NONE`。

认证鲁棒性测试端JSON使用`roundSummary + records`：公共轮次、目标、计划状态和聚合计数只保存一次，三种计划参数保留各自专用字段；CSV为便于统计而在每个逐次事件行重复轮次摘要。测试端不接收业务载荷，因此其`payloadHash`明确写为`NOT_DISTRIBUTED`，不得伪造为节点载荷Hash。源IP字段保存程序当前优选的本机组播接口IPv4，并用`PREFERRED_LOCAL_MULTICAST_INTERFACE`说明其语义。组播仍由操作系统选择实际路由，实机实验应同时结合Receiver观测中的实际UDP源IP复核。

## 主要风险及已实现的处理

- 跨平台线程生命周期：执行器和Sender故障策略均使用可停止、可唤醒、可等待结束的工作线程；停止会清空有界任务缓存。
- Qt跨线程回调：工作线程只通过Qt排队信号更新界面，不在状态锁内执行GUI回调。
- 暂停和统一时间基准：正常节点沿用统一逻辑时间线；测试端与节点启动使用同一个未来毫秒时间戳，并在运行时再次检查截止时间。
- 高频发包：复用单个UDP socket，PPS、持续时间和单包大小有固定上限；详细记录最多保留最近20000条，聚合计数不因淘汰丢失。
- 控制面拥塞：运行状态按250ms节流，但完成、失败和停止状态强制立即回传。
- Receiver资源消耗：接收队列、候选版本和详细异常记录均有固定上限；无效流量先做固定头、来源和上下文检查，并按1000ms窗口汇总。
- 公平性：标签保护分组必须整除每间隔发包数；改进模式自动使用本轮`groupSize`，原生模式可填写同一数值，避免两种模式使用不同可丢位置。
- 数据精度：CSV保留十进制无符号64位值；JSON中的事件ID、chainId、时间戳和64位计数使用字符串保存，避免JSON数字精度损失。

## 自动验收

`tesla_stage10_tests`覆盖：

- 三种计划及其模式专用详情JSON往返；
- 包含密码算法、间隔参数的公开上下文、计划确认、Ready、统一启动时间和执行状态；
- 三种发送侧故障配置、临时来源映射安装/清除和节点确认；
- 丢包与逻辑断链的组末标签保护、固定延迟和安全上限；
- 临时来源映射仅在指定轮次内生效，并在停止后自动清除。
- 逐轮归档JSON往返、角色专用详情，以及真实文本认证结束后Sender/Receiver自动生成一致的载荷SHA-256归档。

Windows审阅方案的完整构建和测试命令为：

```powershell
cmake --preset windows-vs2022-review
cmake --build build/windows-review --config Release
ctest --test-dir build/windows-review -C Release --output-on-failure
```

Ubuntu WSL2使用`BUILD_GUI=OFF`完成公共库、Linux NodeAgent及测试目标构建；Windows与Linux当前均为9/9自动化测试通过。Linux测试额外包含POSIX NodeAgent网络服务测试。

软件实现完成后只剩部署环境验收：在封闭局域网内启动Manager、至少两个广播节点和一个独立测试端，验证真实网卡组播、主机防火墙、节点时钟同步、连续多轮运行以及实际性能数据。该项依赖实机和网络环境，不以模拟结果替代。
