# 阶段5四套GUI框架与连接

阶段5建立四套相互独立的Qt 6 GUI程序。每套程序由一个主窗口模块和一个网络控制模块组成，
避免把界面、TCP分帧、UDP发现和业务状态全部堆入 `main.cpp`，也不为每个小控件拆分独立类。

## GUI职责

| 程序 | 已建立的页面与连接能力 | 本阶段明确不做 |
|---|---|---|
| 集中管理GUI | 节点表、参数与载荷、实验控制、攻击端、文件Hash比较；UDP扫描、心跳跟踪、连接全部、断开全部和状态刷新 | 不发送文本/文件，不启动认证，不显示模拟指标 |
| PC广播节点GUI | 节点状态、报文、异常、密钥链、矩阵、文件和日志；Qt TCP管理服务、UDP发现与心跳 | 不生成Sender认证报文，不接收文件，不生成模拟报文记录 |
| 无人机监控GUI | NodeAgent连接、报文、异常、计算、估算能耗、文件和日志；MONITOR握手、Ping和状态查询 | 不拥有MANAGER权限，不配置Sender或控制实验 |
| 攻击测试端GUI | 管理连接、目标上下文、攻击配置、攻击报文、统计和日志；独立TCP控制服务、ATTACKER发现和TESLA组播只读监听 | 不保存合法载荷，不篡改、重放或发送DoS流量 |

阶段6至阶段10对应的按钮保持禁用并标明接入阶段。界面不使用固定演示数据伪造运行结果。

## 连接和权限

- 集中管理GUI通过临时UDP源端口发送发现请求，避免同机多个服务共享发现端口时响应被错误分配；
- 管理GUI另行监听固定发现端口上的周期心跳，超过3000ms自动关闭对应TCP连接；
- PC节点和NodeAgent使用 `NodeControlMessage`，连接首帧固定为角色 `MANAGER` 或 `MONITOR`；
- 无人机监控GUI固定发送 `MONITOR`，只能使用Ping、状态查询以及后续监控事件；
- 攻击端使用全部带 `ATTACK_` 线缆类型前缀的独立 `AttackControl` 协议，不接受普通节点配置和文件帧；
- 所有Qt局域网TCP客户端显式使用 `NoProxy`，不把热点或局域网地址交给Windows系统代理。

## 多网卡处理

管理GUI向所有活动非回环IPv4接口的广播地址发送扫描，同时保留全局广播和本机测试目标。
同一进程可能通过多个本机接口响应，管理GUI使用“节点角色 + 节点名称 + 管理端口”去重，
并优先选择私有IPv4和物理Wi-Fi/以太网地址。虚拟机、Hyper-V、WSL、VPN和常见隧道接口降低优先级。

PC节点和攻击端也使用相同的物理局域网优先思想生成节点名称。Windows防火墙仍可能在程序首次
监听TCP或UDP时弹出授权提示；程序不会自动修改系统防火墙规则。

## 攻击端组播边界

攻击端启动后加入内部TESLA组播地址并报告真实监听状态。阶段5收到的组播数据立即从Socket缓冲区
读取并丢弃，不解析、不保存、不修改，也不建立攻击源映射。攻击捕获和执行只在阶段10实施。

## 自动验收

`tesla_gui_stage5_tests`使用隔离高端口完成：

1. 四套GUI在 `QT_QPA_PLATFORM=offscreen` 环境构造主窗口并安全退出；
2. 同时启动PC节点服务和攻击端服务；
3. 管理GUI通过UDP发现两者，并分别完成节点控制协议和攻击控制协议连接；
4. 验证PC Receiver状态和攻击端组播监听状态来自真实TCP响应；
5. 无人机GUI以MONITOR身份连接PC节点并取得状态；
6. 测试结束后回收所有子进程。

CTest显式把Qt `bin`目录加入测试进程的 `PATH`。这只解决构建机测试运行环境，不替代发布时的
`windeployqt`部署。
